# Ansible — Nginx webserver for my homepage

Configures the Azure VM (provisioned by **azure-tf-architecture**) as an Nginx webserver on **Ubuntu 24.04 LTS** with **PHP 8.3** (php-fpm). SSH runs over Tailscale and TLS is issued automatically by certbot.

### Related repositories

- [Homepage version 1](https://github.com/Aapok0/homepage)
- [Homepage version 2](https://github.com/Aapok0/homepage-bulma)
- [Azure Terraform architecture](https://github.com/Aapok0/azure-tf-architecture)

## How it works

Four roles, applied in this order:

| Role | Responsibility |
|------|----------------|
| `server_init` | Admin user, SSH hardening (key-only, no root), swap, base packages |
| `tailscale` | Installs Tailscale and joins the tailnet (SSH transport) |
| `firewall` | UFW — 80/443 from anywhere, SSH only on `tailscale0` |
| `nginx` | Deploys the apps and obtains/renews TLS certificates |

Key design points:

- **SSH over Tailscale** — port 22 is never exposed publicly, so a changing home IP can't lock you out.
- **Single-pass TLS** — certbot uses the webroot challenge; TLS turns on automatically once the cert exists and renews via cron with an nginx reload hook.
- **Pinned apps** — each entry in `nginx_apps` is checked out at a `version` (tag/branch/commit) for reproducible deploys.

Playbooks: `server_init.yml` (runs server_init → tailscale → firewall), `nginx.yml`, and standalone `tailscale.yml` / `firewall.yml`.

## Configuration

Copy the committed templates (real files stay gitignored):

| Template | Copy to |
|----------|---------|
| `inventory/production.example` | `inventory/production` |
| `group_vars/servers.example.yml` | `group_vars/servers/main.yml` |
| `group_vars/nginx.example.yml` | `group_vars/nginx.yml` |

`group_vars/servers` is a **directory**: your vars go in `main.yml`; `firewall_allowed_ips.yml` is generated by Terraform's sync script and must not be edited by hand.

**Variables you must set** (full reference lives in each role's README and the `*.example.yml` files):

- `group_vars/servers/main.yml` — `server_init_admin`, `tailscale_kv_name`, `tailscale_tags`, `firewall_ufw_rules`
- `group_vars/nginx.yml` — `nginx_domains`, `nginx_cert_email`, `nginx_apps` (each with a `version`)

> Placeholder IPs like `203.0.113.x` in the `*.example` files are [RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) documentation addresses — replace them.

### Inventory & SSH config from Terraform

After `terraform apply`, generate the inventory, firewall allowlist and SSH config from **azure-tf-architecture**:

```bash
cd azure-tf-architecture
./scripts/sync-ansible-inventory.sh
./scripts/sync-firewall-allowlist.sh
./scripts/sync-ssh-config.sh
```

- The inventory host and SSH `HostName` default to the **Tailscale node name** (the VM / MagicDNS name). Pass `--public-ip` before Tailscale exists, or `--tailnet-suffix <tailnet>.ts.net` for the MagicDNS FQDN.
- `sync-firewall-allowlist.sh` mirrors Terraform's `admin_allowed_ips` into the UFW fallback allowlist.
- Set `ANSIBLE_REPO` if this repo is not a sibling of `azure-tf-architecture`.

## Deploy

### Prerequisites

1. Ansible ≥ 2.15.3 and the pinned collections (`ansible.posix`, `community.general`):
   ```bash
   ansible-galaxy collection install -r requirements.yml
   ```
2. Tailscale configured (one-time) — see [Tailscale setup](#tailscale-setup): create a tailnet, install it on your machine, create an OAuth client + tag, store the key in Key Vault, and set `tailscale_kv_name` / `tailscale_tags`.

### First-time setup (fresh VM)

A new VM has no public SSH rule and no Tailscale yet, so the first pass connects over the public IP and then switches to the tailnet. Run from **azure-tf-architecture** unless noted.

1. Add a temporary public SSH rule:
   ```bash
   ./scripts/bootstrap-ssh-rule.sh
   ```
2. Generate the inventory and SSH config against the public IP:
   ```bash
   ./scripts/sync-ansible-inventory.sh --public-ip
   ./scripts/sync-ssh-config.sh --public-ip
   ```
3. Initialise the server — runs `server_init` → `tailscale` → `firewall`. This creates the admin user, joins the tailnet, then locks SSH to `tailscale0`; your in-flight session survives the UFW reload.
   ```bash
   cd ../homepage-webserver-ansible
   ansible-playbook -i inventory/production server_init.yml --diff --check   # preview
   ansible-playbook -i inventory/production server_init.yml --diff
   ```
   The admin password is prompted **only** when a new user must be created (set `server_init_admin`; supply `server_init_admin_pass` via Vault for unattended runs).
4. Confirm tailnet SSH works: `ssh <tailnet-node-name>`.
5. Re-point inventory/SSH config at the tailnet and remove the temporary rule:
   ```bash
   cd ../azure-tf-architecture
   ./scripts/sync-ansible-inventory.sh
   ./scripts/sync-ssh-config.sh
   ./scripts/remove-ssh-rule.sh
   ```
6. Deploy the site:
   ```bash
   cd ../homepage-webserver-ansible
   ansible-playbook -i inventory/production nginx.yml --diff --check   # preview
   ansible-playbook -i inventory/production nginx.yml --diff
   ```

> Prefer not to open public SSH at all? Skip step 1's temporary rule and bring Tailscale up directly via `az vm run-command`, then run Ansible over the tailnet (sync with defaults instead of `--public-ip` in step 2):
>
> ```bash
> az vm run-command invoke -g <rg> -n <vm> --command-id RunShellScript \
>   --scripts "curl -fsSL https://tailscale.com/install.sh | sh && tailscale up --authkey tskey-XXXX --advertise-tags=tag:server"
> ```
>
> The Azure Serial Console (portal → VM → Serial console) is an interactive alternative for the same purpose.

### Routine deploy

Tailscale is already up, so sync with defaults and run the relevant playbook:

```bash
# azure-tf-architecture — only if infra/IPs changed
./scripts/sync-ansible-inventory.sh && ./scripts/sync-firewall-allowlist.sh && ./scripts/sync-ssh-config.sh

# homepage-webserver-ansible
ansible-playbook -i inventory/production nginx.yml --diff         # redeploy site / certs
ansible-playbook -i inventory/production server_init.yml --diff   # only if server/firewall config changed
```

## Tailscale

### Tailscale setup

One-time, before the first deploy:

1. Create a tailnet at [tailscale.com](https://tailscale.com) (free plan is fine).
2. Install Tailscale on every device you SSH from and log in:
   ```bash
   sudo tailscale up
   sudo tailscale set --operator=$USER   # once, so you can use tailscale/ssh without sudo
   ```
3. Create an **OAuth client** (Settings → OAuth clients) with the `auth_keys` write scope and a tag (e.g. `tag:server`); define/grant that tag in the ACL policy. OAuth-backed nodes can have key expiry disabled — plain auth keys always expire.
4. Store the secret in Key Vault (kept out of Terraform state, like the SSH private key):
   ```bash
   az keyvault secret set --vault-name <kv> --name homepage-tailscale-oauthkey --value tskey-client-XXXX
   ```
5. Point the role at it in `group_vars/servers/main.yml`:
   ```yaml
   tailscale_kv_name: <kv>            # e.g. sdc-prd-homepage-kv
   tailscale_tags: ['tag:server']     # tailscale_secret_name defaults to homepage-tailscale-oauthkey
   ```
   (Or skip `tailscale_kv_name` and `export TAILSCALE_AUTHKEY=...` yourself.)

The VM joins the tailnet during `server_init.yml` (or run `tailscale.yml` standalone). Afterwards, **disable key expiry** for the node in the admin console so it never drops off; optionally enable MagicDNS.

### Migrating an existing VM (avoid lockout)

Do this in order so you never lose access:

1. Run `tailscale.yml` over your current public-IP SSH and confirm `ssh <user>@<node-name>` works.
2. Re-run the sync scripts with their defaults (tailnet name).
3. `terraform apply` in **azure-tf-architecture** (removes the public SSH NSG rule).
4. Run `firewall.yml` (locks UFW SSH to `tailscale0`). Established sessions survive; new ones use the tailnet.

### Recovery & escape hatch

- Tailnet unreachable? Fix the host without SSH via the Azure Serial Console or:
  ```bash
  az vm run-command invoke -g <rg> -n <vm> --command-id RunShellScript \
    --scripts "tailscale up --authkey tskey-auth-XXXX"
  ```
- To temporarily disable Tailscale-only SSH, set `firewall_ssh_over_tailscale_only: false` in `group_vars/servers/main.yml` — it falls back to the `firewall_allowed_ips` allowlist (synced from Terraform `admin_allowed_ips`).

## Reference

### Sync scripts (in azure-tf-architecture)

- `sync-ansible-inventory.sh` — rewrites `inventory/{development,test,production}` from state; hosts no longer in state are dropped.
- `sync-firewall-allowlist.sh` — writes `group_vars/servers/firewall_allowed_ips.yml` from `admin_allowed_ips`.
- `sync-ssh-config.sh` — rewrites only the `# BEGIN/END terraform-managed:` blocks in `~/.ssh/config`; other entries are left alone.
- All accept `--public-ip` and `--tailnet-suffix <suffix>` (also honoured as `*_USE_TAILSCALE` / `TAILNET_SUFFIX` env vars). Use `--public-ip` before Tailscale is installed.

### Certificate flow

`nginx.yml` serves the ACME `http-01` challenge from `nginx_acme_webroot` on port 80, runs `certbot certonly --webroot`, then re-renders the vhost with the 443 blocks — all in a single run. TLS is enabled automatically once the cert exists, and a daily cron renews it with `--deploy-hook 'systemctl reload nginx'`. Set `nginx_manage_certs: false` to skip certbot (local/dev). A `--check` run can't issue a cert; it shows the HTTP-only vhost.

### SSH hardening

`server_init` disables password and root login, including a `00-hardening.conf` sshd drop-in that overrides cloud-image defaults. Connect as your non-root admin user.
