# Ansibles to setup an Nginx webserver for my homepage

The VM that the webserver is deployed to is hosted in Azure (**azure-tf-architecture**) on **Ubuntu 24.04 LTS** with **PHP 8.3** (php-fpm). Pin the image version in Terraform; PHP package names are set in the nginx role defaults (`nginx_php_fpm_package`).

The way the roles have been written is a compromise between between the needs of my project and wanting to write reusable Ansible code. The roles could have more features and configurability, but any more would be unnecessary for my needs. Certbot uses the **webroot** challenge and TLS is enabled automatically once a certificate exists, so a single `nginx.yml` run bootstraps a fresh host. See [Usage](#Usage).

App checkouts are **pinned** via a per-app `version` (tag/branch/commit) in `nginx_apps` for reproducible deploys.

## Configuration

Committed templates (copy and edit locally; real files stay gitignored):

| Template | Copy to |
|----------|---------|
| `inventory/production.example` | `inventory/production` |
| `group_vars/servers.example.yml` | `group_vars/servers/main.yml` |
| `group_vars/nginx.example.yml` | `group_vars/nginx.yml` |

> `group_vars/servers` is a **directory**: hand-maintained vars go in `main.yml`; `firewall_allowed_ips.yml` is generated by the Terraform sync script (below) and must not be edited by hand.

### Terraform handoff

After `terraform apply` in the **azure-tf-architecture** repository:

> **Placeholders:** IP addresses like `203.0.113.x` in this README and in `*.example` files are [RFC 5737 documentation addresses](https://datatrfc.ietf.org/doc/html/rfc5737) — not real public IPs. Replace them with your VM IP and home IP after deploy.

1. **Host** — Ansible connects over Tailscale, so `./scripts/sync-ansible-inventory.sh` writes the **tailnet node name** (the VM/MagicDNS name) as the inventory host, not the public IP. Pass `--public-ip` (before Tailscale exists) or `--tailnet-suffix <tailnet>.ts.net` (FQDN) to change this. `terraform output public_ip_info_out` still shows the public IP for reference.
2. **ansible_user** — generated per VM; run `terraform output admin_user_out` or use `./scripts/sync-ansible-inventory.sh` after apply (reads **azure-tf-architecture** `ansible_hosts_out`).
3. **SSH access** — SSH is reached over **Tailscale**, not a public IP (see [Tailscale](#tailscale-ssh-access) below). There is no public SSH NSG rule and UFW only allows port 22 on `tailscale0`. `admin_allowed_ips` in **azure-tf-architecture** still feeds the NSG ICMP rule and is mirrored to `group_vars/servers/firewall_allowed_ips.yml` as the UFW SSH **fallback** (used only if `firewall_ssh_over_tailscale_only: false`). Do not edit the allowlist in Ansible by hand.
4. **DNS** — **azure-tf-architecture** creates the Azure DNS zone and A records; point your registrar nameservers at Azure if not already done.
5. **Certificates** — set `nginx_cert_email` in `group_vars/nginx.yml`; domains must resolve to the VM before the first `nginx.yml` run. TLS turns on automatically once the cert is issued (set `nginx_manage_certs: false` to skip certbot).

After `terraform apply`, sync inventory from the **azure-tf-architecture** repository (rewrites managed inventory files; removes hosts no longer in state):

```bash
cd azure-tf-architecture
./scripts/sync-ansible-inventory.sh
./scripts/sync-firewall-allowlist.sh   # writes group_vars/servers/firewall_allowed_ips.yml from admin_allowed_ips
./scripts/sync-ssh-config.sh           # optional: replaces terraform-managed ~/.ssh/config blocks only
```

`sync-ansible-inventory.sh` regenerates `inventory/development`, `inventory/test`, and `inventory/production` every run. `sync-firewall-allowlist.sh` mirrors `admin_allowed_ips` into the Ansible firewall allowlist so UFW and NSG stay in sync. `sync-ssh-config.sh` strips blocks between `# BEGIN terraform-managed:` / `# END terraform-managed:` markers and rewrites them from state; other SSH config entries are left alone.

Because SSH goes over Tailscale, `sync-ssh-config.sh` sets `HostName` to the **tailnet node name** (the VM/MagicDNS name), not the public IP — so MagicDNS must be enabled and your client on the tailnet. Pass `--tailnet-suffix <tailnet>.ts.net` to use the MagicDNS FQDN (more robust), or `--public-ip` to fall back to the public IP (only works while a public SSH rule exists, e.g. during bootstrap). Both scripts also honour the `SSH_USE_TAILSCALE` / `ANSIBLE_USE_TAILSCALE` / `TAILNET_SUFFIX` environment variables; the flags override them.

> Before the **first** `server_init` run, Tailscale isn't installed yet — generate the inventory and SSH config with `--public-ip` so they target the public IP reachable via the temporary NSG rule. Switch to the defaults once Tailscale is up.

Set `ANSIBLE_REPO` if **homepage-webserver-ansible** is not a sibling directory.

### Inventory

For the servers you want to run Ansible against, create an inventory file under `inventory/` (for example `inventory/production`) with these groups. Use your real VM IP — not the placeholder below:

```ini
[servers]
203.0.113.20 ansible_user=your_admin_user

[nginx]
203.0.113.20 ansible_user=your_admin_user
```

Variables for the roles go in `group_vars/` named after inventory groups (`servers`, `nginx`). See the `*.example.yml` files for a working homepage setup. The roles also support the following variables.

server_init:

```yaml
# If you want to create a new admin user, defaults to <user_home>/.ssh/id_rsa.pub
server_init_admin_pubkey: /path/to/ssh_pubkey
# Extra packages you want, by default following are installed: cron, curl, git, grep, less, rsync, unattended-upgrades and vim
server_init_extra_packages:
  - fzf
  - htop
  - ncdu
  - ripgrep
  - tmux
# Whether unattended-upgrades is enabled, default is true
server_init_auto_upgrades: true
# Swap space you need in the server in bytes
server_init_swapfile_size: 1500000000
# Tendency to move processes from physical memory to swap space
server_init_swappiness: 60
```

firewall:

```yaml
# Whether IPV6 is disables in the server, default is false
firewall_disable_ipv6: true
# SSH only over the Tailscale interface (default). false = use firewall_allowed_ips.
firewall_ssh_over_tailscale_only: true
firewall_ssh_interface: tailscale0
# Which ports are open, removing won't remove rules (need to do it manually)
firewall_ufw_rules:
  - port: 80
    proto: tcp
  - port: 443
    proto: tcp
```

SSH is allowed only on `tailscale0` by default (see [Tailscale](#tailscale-ssh-access)). `firewall_allowed_ips` is the **fallback** used when `firewall_ssh_over_tailscale_only: false`; it is generated into `group_vars/servers/firewall_allowed_ips.yml` from Terraform's `admin_allowed_ips` by `azure-tf-architecture/scripts/sync-firewall-allowlist.sh`, not edited by hand. The role default is `any` so the fallback never locks you out.

nginx:

```yaml
---
# Configuration -> most of the following are the default values
nginx_user: nginx
nginx_worker_processes: auto

## Configuration - Events
nginx_worker_connections: 1024
nginx_events_use: epoll # if not set, nginx will set according to server's OS
nginx_events_multi_accept: 'on'

## Configuration - HTTP
### HTTP - Basic Settings
nginx_sendfile: 'on'
nginx_sendfile_max_chunk: 2m
nginx_tcp_nopush: 'on'
nginx_types_hash_max_size: 1024
nginx_server_tokens: 'off'
nginx_server_name_in_redirect: 'on'
server_names_hash_bucket_size # if not set, nginx will set default according to server
nginx_default_type: application/octet-stream

### SSL Settings
nginx_ssl_protocols: 'TLSv1.2 TLSv1.3'
nginx_ssl_prefer_server_ciphers: 'on'
nginx_ssl_ciphers: 'HIGH:!aNULL:!MD5' # if not set, nginx will set default

### Logging Settings
nginx_error_debug_level: error

### Gzip Settings
nginx_gzip: 'on'
nginx_gzip_vary: 'off'
nginx_gzip_proxied: 'off' # if not set, nginx will set default
nginx_gzip_comp_level: 1
nginx_gzip_buffers: '16 8k'
nginx_gzip_http_version: 1.1
nginx_gzip_types: 'text/html text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript'

# Vhost
# TLS is enabled automatically once a cert exists; set false to skip certbot (e.g. local/dev)

# Webroot served for the ACME http-01 challenge (default below)
nginx_acme_webroot: /var/www/acme
nginx_domains:
  - www.example.com # First in the list will be used as main domain
  - example.com # The rest of the domains will be redirected to main domain
nginx_cert_email: name@example.com
nginx_listen_ipv6: false
nginx_security_headers_enabled: true
nginx_security_headers_hsts: 'max-age=31536000; includeSubDomains'
nginx_security_headers_csp: "default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self'; font-src 'self'; frame-ancestors 'none'"
nginx_apps:
  - name: example
    main: true # App that will be served when using root of the domain
    repo: https://github.com/Aapok0/homepage.git
    version: main # Pin a tag (e.g. v1.2.0), branch, or commit for reproducible deploys
    location: example # Path of app -> www.example.com/example/
    npm: true # The tasks use pnpm
    sass: true
    php: true
    build_command: css-build # pnpm build command
  - name: example2
    main: false
    repo: https://github.com/Aapok0/homepage-bulma.git
    version: v2.0.0
    location: example2 # Path of app -> www.example.com/example2/
    npm: false
    sass: false
    php: true
```

`version` is passed to `ansible.builtin.git`; if omitted it defaults to `main`. App build/install steps only run for apps whose checkout actually changed.

## Usage

### Required collections

Install pinned collections from `requirements.yml`:

```bash
ansible-galaxy collection install -r requirements.yml
```

Collections: `ansible.posix`, `community.general`

### Steps

```bash
# Requires Ansible version 2.15.3 or above and collections above

# Clone repository, install collections, copy *.example files (see Configuration)

cp inventory/production.example inventory/production
mkdir -p group_vars/servers
cp group_vars/servers.example.yml group_vars/servers/main.yml
cp group_vars/nginx.example.yml group_vars/nginx.yml
# Edit inventory IP, nginx_cert_email, nginx_domains, nginx_apps versions
# (firewall_allowed_ips is generated by the Terraform sync script — see below)

# From azure-tf-architecture, sync inventory + firewall allowlist. On the FIRST
# run Tailscale isn't up yet, so generate the inventory against the public IP:
#   ./scripts/sync-ansible-inventory.sh --public-ip   # drop --public-ip after Tailscale is up
#   ./scripts/sync-firewall-allowlist.sh
# See the "Fresh VM bootstrap" section for the full first-run order.

# Set tailscale_kv_name in group_vars/servers/main.yml so the tailscale role pulls
# the OAuth key from Key Vault via az (or export TAILSCALE_AUTHKEY instead).
# server_init.yml runs the tailscale role before the firewall locks SSH to tailscale0.

# Run server_init.yml playbook (first with check and then without, if evertyhing looks fine)
# Runs server_init -> tailscale -> firewall.
# Set server_init_admin in group_vars (recommended). A password is only prompted/
# needed when a NEW admin user must be created (i.e. you are not already connected
# as that user); supply server_init_admin_pass (Vault) for unattended runs.
ansible-playbook -i inventory/inventory_file server_init.yml --diff --check
ansible-playbook -i inventory/inventory_file server_init.yml --diff
```

### Certificate bootstrap (single run)

No manual toggle is needed. On a fresh host:

1. Make sure `nginx_domains` resolve to the VM (DNS A records) and ports 80/443 are open.
2. Run `nginx.yml`. The role renders an HTTP vhost that serves the ACME `http-01` challenge from `nginx_acme_webroot`, reloads nginx, runs `certbot certonly --webroot`, then re-renders the vhost with the 443 server blocks and reloads.
3. Subsequent runs are idempotent: the cert is detected (`creates` guard), TLS stays on, and `certbot renew -q --deploy-hook 'systemctl reload nginx'` (daily cron) renews and reloads automatically.

```bash
# First run with --check to preview, then for real
ansible-playbook -i inventory/inventory_file nginx.yml --diff --check
ansible-playbook -i inventory/inventory_file nginx.yml --diff
```

> The `--check` run cannot actually issue a cert; on a truly fresh host it will show the HTTP-only vhost. The real run performs issuance and TLS enablement in one pass.

## Tailscale (SSH access)

SSH is reached over a [Tailscale](https://tailscale.com) tailnet instead of a public IP, so a changing home IP never locks you out and port 22 is never exposed to the internet. The `tailscale` role installs Tailscale and joins the host; the `firewall` role then only allows port 22 on the `tailscale0` interface (`firewall_ssh_over_tailscale_only: true`).

### One-time setup

1. **Create a tailnet** at [tailscale.com](https://tailscale.com) (free plan is fine).
2. **Install Tailscale on every device you SSH from** (laptop, etc.) and log in:
   ```bash
   sudo tailscale up
   sudo tailscale set --operator=$USER   # once, so you can use tailscale/ssh without sudo
   ```
3. **Create an OAuth client** in the admin console (Settings → OAuth clients) with the `auth_keys` write scope and a tag (e.g. `tag:server`). Use an OAuth client secret rather than a plain auth key because **only OAuth-backed nodes can have key expiry disabled** — plain auth keys always expire. Define/grant the tag in the tailnet ACL policy.
4. **Store the secret in Key Vault** (kept out of Terraform state, like the SSH private key):
   ```bash
   az keyvault secret set --vault-name <kv> --name homepage-tailscale-oauthkey --value tskey-client-XXXX
   ```
5. **Configure the role** in `group_vars/servers/main.yml` so it pulls the secret from KV (via the `az` CLI on your machine) and advertises the tag:
   ```yaml
   tailscale_kv_name: <kv>            # e.g. sdc-prd-project-kv
   tailscale_tags: ['tag:server']
   # tailscale_secret_name defaults to homepage-tailscale-oauthkey
   ```
   (Alternatively skip `tailscale_kv_name` and `export TAILSCALE_AUTHKEY=...` yourself.)
6. **Join the VM:**
   ```bash
   ansible-playbook -i inventory/production tailscale.yml --diff
   ```
7. In the admin console, **disable key expiry** for the VM node so it never drops off the tailnet. (Optional: enable MagicDNS.)
8. **Point Ansible/SSH at the tailnet address.** Re-run `sync-ssh-config.sh` (it now writes `HostName <tailnet node name>` instead of the public IP), and set `ansible_host` in the inventory to the node's MagicDNS name (the VM name) or its `100.x.y.z` address, since the public IP no longer accepts SSH:
   ```ini
   [servers]
   sdc-prd-homepage-webserver-vm-0 ansible_user=<user>
   ```
   Requires MagicDNS enabled. If bare-name resolution doesn't work on your client, use the FQDN (`TAILNET_SUFFIX=<tailnet>.ts.net ./scripts/sync-ssh-config.sh`).

### Migrating the existing VM (avoid lockout)

Do this **in order** so you never lose access:

1. Run `tailscale.yml` over your current public-IP SSH and confirm `ssh <user>@<magicdns-name>` works.
2. Switch the inventory/SSH config to the tailnet name: re-run `sync-ansible-inventory.sh` and `sync-ssh-config.sh` with their defaults.
3. `terraform apply` in **azure-tf-architecture** (removes the public SSH NSG rule).
4. Run `firewall.yml` (locks UFW SSH to `tailscale0`). Established sessions survive; new ones must use the tailnet.

### Fresh VM bootstrap

A brand-new VM has **no public SSH rule** and Tailscale isn't installed yet, so the first pass must connect over the public IP. Use a temporary NSG rule and the `--public-ip` flag, then switch to the tailnet once Tailscale is up.

Prerequisite: the OAuth key is in Key Vault and `tailscale_kv_name`/`tailscale_tags`/`server_init_admin` are set (steps above), or the `tailscale` role will fail.

```bash
cd azure-tf-architecture

# 1. Temporary public SSH rule (source = admin_allowed_ips, or any if empty)
./scripts/bootstrap-ssh-rule.sh

# 2. Generate inventory + ssh config against the PUBLIC IP for this first pass
./scripts/sync-ansible-inventory.sh --public-ip
./scripts/sync-ssh-config.sh --public-ip

# 3. Run server_init.yml: server_init -> tailscale (brings the tailnet up) ->
#    firewall (locks SSH to tailscale0). The in-flight public-IP session survives
#    the UFW reload (conntrack), so the play finishes.
cd ../homepage-webserver-ansible
ansible-playbook -i inventory/production server_init.yml --diff

# 4. Verify tailnet SSH works
ssh <tailnet-node-name>

# 5. Re-generate against the tailnet (defaults) and drop the temp rule
cd ../azure-tf-architecture
./scripts/sync-ansible-inventory.sh
./scripts/sync-ssh-config.sh
./scripts/remove-ssh-rule.sh

# 6. Deploy the site over the tailnet
cd ../homepage-webserver-ansible
ansible-playbook -i inventory/production nginx.yml --diff
```

(The Azure **Serial Console** / `az vm run-command` is an alternative to the temp rule if you'd rather not open public SSH at all.)

### Recovery

If the tailnet is unreachable, fix the host without SSH via the Azure Serial Console or:

```bash
az vm run-command invoke -g <rg> -n <vm> --command-id RunShellScript \
  --scripts "tailscale up --authkey tskey-auth-XXXX"
```

### Escape hatch

Set `firewall_ssh_over_tailscale_only: false` in `group_vars/servers/main.yml` to fall back to the `firewall_allowed_ips` allowlist (synced from Terraform `admin_allowed_ips`).
